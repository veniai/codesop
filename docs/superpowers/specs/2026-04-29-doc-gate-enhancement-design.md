# codesop 文档判定 Gate 增强 Spec

> **Date**: 2026-04-29
> **Scope**: 增强文档判定 gate，从通用判定标准升级为变更类型驱动的精确映射 + 自检清单，减少漏改
> **Inspiration**: KKKKhazix/neat-freak（变更影响矩阵 + 自检清单）
> **Principle**: 不新增 skill，只强化现有 gate；保持 concise
> **Codex Review**: 3 条修订已合并（表头覆盖删除、新增配置文件行、自检第 5 条对齐矩阵）

---

## 0. 问题陈述

当前文档判定 gate 有两个弱点：

1. **判定标准太泛** — "技术架构变化时更新 CLAUDE.md" 没有告诉 AI "新增了环境变量 → CLAUDE.md 环境变量表 + README 配置段"。AI 要自己推断哪些文档的哪些段落受影响，推断经常不够精确，导致漏改。
2. **没有自检** — gate 输出"未更新"后就结束了，没有回头看"这次改了 X，X 相关的文档段落真的检查了吗？"。neat-freak 用强制 checklist 解决这个问题。

## 1. 改什么（2 件事 + SKILL.md 流程更新）

### 1.1 变更影响矩阵

**位置**: `templates/system/AGENTS.md` 文档判定段

**当前**: 4 条通用判定标准（CLAUDE.md 改什么、PRD.md 改什么...）

**改为**: 保留通用标准，追加一张**变更类型 → 必检文档**映射表。AI 判定文档时先查这张表，再结合通用标准做最终判断。

矩阵管范围（哪些文档的哪些段落需要检查），通用标准管阈值（检查后是否真的需要更新）。不在表中的变更类型，回退到通用判定标准。

```markdown
**变更影响矩阵**（判定时按变更类型查表）：

| 变更类型 | CLAUDE.md | PRD.md | README.md | CONTEXT.md |
|---------|-----------|--------|-----------|------------|
| 新增/改/删命令或脚本 | 命令段 | - | 安装/运行段 | - |
| 新增/改/删目录或文件 | 架构树 | - | - | - |
| 新增/改/删环境变量 | 环境段 | - | 配置段 | - |
| 新增/改/删配置文件 | 架构段 | - | 配置/安装段 | - |
| 新增/改/删依赖 | 依赖段 | - | 安装/兼容段 | - |
| 新增/改/删 API/接口 | 接口段 | - | 接口段 | - |
| 新增/改/删架构决策 | 架构段 | 决策记录 | - | - |
| 新增/改/删产品功能 | - | 功能/范围/进度 | - | - |
| 新增/改/删领域术语 | - | - | - | 术语表 |
| 新增/改/删安装/部署 | 命令段 | - | 安装/部署段 | - |
| 新增/改/删测试 | 命令段 | - | 测试段 | - |

"-" = 该变更类型通常不影响该文档。多行匹配时，所有标记的文档都要检查。
CONTEXT.md 仅在文件存在时判定，不在表中出现的变更类型默认不影响。
不在表中的变更类型，回退到通用判定标准。
```

**设计决策**:
- 表格形式，不是自由文本 — 结构化，AI 能直接查表
- "-" 表示通常不影响，不是禁止 — AI 仍可基于通用标准判断
- 只列高频变更类型，不全覆盖 — YAGNI，后续按需扩展
- 表头用"新增/改/删"覆盖删除场景 — 删除命令/目录同样需要更新文档

### 1.2 自检清单

**位置**: `templates/system/AGENTS.md` 文档判定段（矩阵之后）

**追加内容**:

```markdown
**自检清单**（判定输出前逐条确认，任一项不通过则补检）：

- 本次新增/改/删的文件/目录 → CLAUDE.md 架构树是否已同步？
- 本次新增/改/删的环境变量 → README.md 配置段是否已提及？
- 本次新增/改/删的命令/脚本 → CLAUDE.md 命令段 + README.md 安装段是否已同步？
- 本次新增/改/删的依赖 → CLAUDE.md + README.md 是否都已更新？
- 本次新增/改/删的 API/接口 → CLAUDE.md 接口段 + README.md 接口段是否已同步？

全部通过 → 判定完成。不通过 → 回到判定步骤补检。
```

**设计决策**:
- 5 条硬检查，覆盖最高频漏改场景，与矩阵对齐
- 不要求输出自检结果 — 内部用，用户看到的是最终判定
- 不与文档判定输出格式冲突 — 自检在输出前执行
- 与 claude-md-management 的关系：gate 是决策点（要不要改），claude-md-management 是执行工具（怎么改）。增强 gate 精度 → 触发 claude-md-management 更准确

### 1.3 SKILL.md §5 gate 流程更新

**位置**: `SKILL.md` §5 Completion Gate

**当前流程**: 3 步（判定 → 调 claude-md-management → 输出格式）

**改为**: 4 步，在判定和输出之间插入矩阵查询 + 自检：

```markdown
1. 识别本次变更类型（对照变更影响矩阵）
2. 按矩阵 + 通用标准对每个文档逐个判断（不允许跳过或批量标记），决定是否需要更新
3. 执行自检清单，逐条确认无漏改
4. 输出判定结果（格式不变）
```

输出格式**不变**，仍然固定为：
```
- CLAUDE.md: 已更新 / 未更新，原因：...
- PRD.md: 已更新 / 未更新，原因：...
- README.md: 已更新 / 未更新，原因：...
```

## 2. 执行计划

2 个文件改动，1 个 commit。

| 文件 | 改什么 |
|------|--------|
| `templates/system/AGENTS.md` | 文档判定段：通用标准后追加变更影响矩阵（11 行表格 + 3 行说明）+ 自检清单（5 条 + 1 行结论） |

## 4. 风险

| 风险 | 缓解 |
|------|------|
| 矩阵表格在 AGENTS.md 中占行过多 | 11 行表格 + 5 条自检 ≈ 20 行，文档判定段从 ~17 行增至 ~37 行，全文从 ~90 行增至 ~110 行，可控 |
| AI 不查表直接跳过通用标准 | 流程写明"先查表再结合通用标准"，SKILL.md §5 也有步骤引用 |
| 自检清单变成形式主义 | 只 5 条，每条对应具体变更类型，与矩阵对齐 |
| 矩阵与通用标准职责模糊 | 说明明确：矩阵管范围，通用标准管阈值 |
