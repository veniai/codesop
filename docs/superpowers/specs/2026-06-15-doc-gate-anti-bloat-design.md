# codesop 文档判定 Gate 防膨胀（减法）纪律 Spec

> **Date**: 2026-06-15
> **Scope**: `templates/system/AGENTS.md` 文档职责 + 文档判定段（+ `SKILL.md §5` 自检可选）
> **Inspiration**: KKKKhazix/neat-freak（减法三原则 + 净涨幅阈值 + 反向删除映射）
> **Relation**: 补全 `2026-04-29-doc-gate-enhancement-design.md`——那条提取了 neat-freak 的正向"补漏"（变更影响矩阵 + 自检清单），本条提取反向"减法"
> **Principle**: 不新增 skill；不破坏现有高风险映射语义；减法与加法并列、不替换；保持 concise

---

## 0. 问题陈述

当前文档判定 gate 只有**正向"补漏"**：高风险映射（变更类型 → 该改哪些文档）判断"该往哪些文档加什么"，缺**反向"减法"**——没有"什么该从文档删"。

后果：CLAUDE.md 只增不减，三类垃圾堆积——
1. **历史叙事**（"X 起 Y 上线，详见 docs/Z"）当变更日志用
2. **一次性事故复盘 / bug fix 流水账**
3. **已被新版本取代的中间态**（"5/6 改 X，5/8 又改 Y"）

这些把真正"下次写代码必须看到的规则"挤出 prompt 重点段，AI 越用越钝——不是模型变笨，是规则手册脑腐了。

neat-freak 的解法是一套**减法纪律**：一条判断标准 + 三原则 + 净涨幅阈值 + 反向删除映射。正向补漏已在 doc-gate-enhancement 落地，本 spec 补反向。

## 1. 改什么（3 件事，集中在 AGENTS.md 两段）

### 1.1 减法判断标准 + 三原则（`## 文档职责` → 防膨胀规则扩展）

**位置**：`templates/system/AGENTS.md`，现有 `**防膨胀规则**` 三条之后追加。

**追加文本**：

```markdown
**减法纪律**（防只增不减——与上方"放哪"互补，管"删啥"）：
- 判断标准：一条信息"下次写代码时看不到会犯错吗？"——不会就删或迁出 CLAUDE.md
- 三原则：减优于加 / 合并优于追加（新信息是对旧信息的更新就改旧条目，加前先 grep 同关键字）/ 删除优于保留（完成的临时计划、推翻的决策、已被取代的中间态、单次事故流水账 → 删）
- CLAUDE.md 是规则手册不是变更日志：历史叙事、事故复盘、bug fix 流水账不进 CLAUDE.md，归 git log / CHANGELOG / docs
```

**设计决策**：
- 挂在现有"防膨胀规则"下，不新开章节——"放哪"和"删啥"是同一件事的两面，分开会割裂
- 判断标准一句话可机械套用（"看不到会犯错吗"），不靠主观
- 三原则有明确动作（grep、改旧条目、删），不是口号

### 1.2 净涨幅红线（`## 文档判定` → 判定原则追加）

**位置**：`templates/system/AGENTS.md`，`判定原则：` 列表末尾追加一条。

**追加文本**：

```markdown
- 净涨幅警戒：本次对 CLAUDE.md 的净改动（增−删，可用 `git diff --numstat <file>` 估）> 30 行时回头审——多半在写历史叙事/便条而非规则。能删先删，不能删迁 docs，剩下的才是规则
```

**设计决策**：
- 30 行是经验阈值（借自 neat-freak），作用是触发"回头审"，不是硬阻断
- 给出估算手段（git diff --numstat）但不写死命令——gate 是决策点不是脚本
- 只管 CLAUDE.md（规则手册），PRD/README 不设此线（它们的膨胀由各自章节规则管）

### 1.3 反膨胀清理清单（`## 文档判定` → 高风险映射之后，独立表）

**位置**：`templates/system/AGENTS.md`，现有 `高风险映射` 之后追加。**独立成表，不塞进高风险映射**——后者语义是"变更类型 → 必检文档"（正向补漏），反向删除是"文档已有内容 → 该不该留"，混表会破坏映射的查表语义。

**追加文本**：

```markdown
反膨胀清理清单（判定时顺带扫描文档已有内容，命中则删/迁，与高风险映射方向相反）：

| 文档中已有的 | 处理 |
|---|---|
| "X 起 Y 上线，详见 docs/Z" 形式历史叙事 | 删——指针角色由速查表承担，叙事归 git log / CHANGELOG |
| 抄自 docs/ 的详细机制 / 数据流 / 公式 | 删——CLAUDE.md 只留边界规则，AI 改到这块自然读 docs |
| 一次性事故复盘（"X 时 Y 挂了 30min"） | 留 ≤1 行红线规则，详情归 docs/PLAYBOOK 或删 |
| 已被新版本取代的中间态（"5/6 改 X，5/8 又改 Y"） | 只留最终态，中间历史删 |
| 相对时间（今天 / 昨天 / 最近 / today / yesterday） | 转绝对日期，或删 |
```

**设计决策**：
- 独立表，5 条覆盖 neat-freak sync-matrix 反向部分的高频项，不全搬（YAGNI）
- "处理"列给明确动作（删/迁/留几行），不只是诊断
- 与 doc-gate-enhancement 的正向矩阵互不干扰

## 3. 不做什么

| 不做 | 为什么 |
|------|--------|
| 新增 neat/sync 独立 skill | 文档判定 gate 已存在，强化比新增好（与 doc-gate-enhancement 同决策） |
| patch claude-md-management | 避免上游 rebase 负担；减法纪律先沉淀在 gate 决策层，验证有效再考虑进工具 |
| 把反向删除塞进高风险映射 | 破坏映射"变更类型→必检文档"的查表语义，方向相反须独立 |
| 记忆系统清理规则 | memory 有自己的规则，不混进文档 gate（与 doc-gate-enhancement 同决策） |
| 净涨幅阈值推广到 PRD/README | 两文档膨胀由各自章节规则管，设线反而干扰 |
| 全量搬运 neat-freak sync-matrix | YAGNI，5 条覆盖高频反膨胀场景 |

## 4. 风险

| 风险 | 缓解 |
|------|------|
| "看不到会犯错吗"判断主观化 | 三原则给明确动作兜底（grep、改旧条目、删），不只靠主观判断 |
| 净涨幅 30 行阈值不适配所有项目 | 作用是"触发回头审"非硬阻断；项目可在项目级 CLAUDE.md 覆盖 |
| 反膨胀清单变成每次强行扫描拖慢 gate | 5 条快速扫，与正向自检并列执行，不额外开流程 |
| 减法误删仍在生效的规则 | 判断标准"看不到会犯错吗"是删的前置条件；不确定时迁 docs 而非直删 |
| AGENTS.md 行数增长 | 净增约 12 行（4 + 1 + 表 7），文档判定段可控 |

## 5. 验收标准

1. `templates/system/AGENTS.md` 文档职责段含"减法纪律"（判断标准 + 三原则 + 规则手册定位）
2. 文档判定段含净涨幅红线（>30 行回头审）
3. 文档判定段含独立的反膨胀清理清单表（5 条），与高风险映射并列且语义不混
4. `bash setup --host claude` 成功同步到全局 `~/.claude/CLAUDE.md`
5. 可选：`tests/detect-environment.sh` 加断言验证"减法纪律""净涨幅""反膨胀"关键词存在
6. `bash tests/run_all.sh` 全部通过
